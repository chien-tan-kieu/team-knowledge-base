How do we rotate secrets for a service in production? I want to know the approval steps, not just the mechanics.
K Knowledge Base · grounded in 3 wiki pages

Secret rotation in production follows a two-person approval flow documented in the security runbook. The rotating engineer opens a change ticket, a second reviewer from the on-call roster signs off, and only then does the rotate-secret CLI proceed to write into Vault.1secret-rotationSecret rotation runbook"All production rotations require a second approver from the security on-call rotation. The approval is logged to the audit stream before the rotation proceeds."Open page →

Three things are worth emphasizing before you start:

  • The change must be filed at least 30 minutes before rotation so on-call has time to review.1secret-rotationSecret rotation runbook"Changes filed with less than 30 minutes lead time are auto-deferred to the next on-call handoff."Open page →
  • Services with active consumers need a dual-read window — the old and new secret are both accepted for at least one deploy cycle.2deploy-pipelineDeploy pipeline"Zero-downtime rotation: the service reads both the active and the staged secret during the migration window. See `dual-read` config flag."Open page →
  • If the secret is third-party (Stripe, SendGrid, etc.), the vendor rotation happens first, and only afterwards does Vault get the new value.3vendor-credentialsVendor credentials"Third-party secrets must be rotated at the source of truth before being written to Vault. The vendor rotation timestamp is the canonical one."Open page →

In short: ticket → second approval → dual-read deploy → rotate. If you want, I can walk you through a specific service's rotation window next.

Deploy Pipeline

Updated 3 days ago Compiled by AI · reviewed by Hien architecture ci-cd ops

Every change goes through the same four-stage pipeline — build, test, stage, promote — with a dual-read window at the promote step so that secret and schema migrations never take the service down.

Overview

The pipeline is defined once in pipelines.yaml and runs identically across every service in the monorepo. Each stage has an explicit contract: an input artifact, a verification step, and a gated promotion to the next stage. No stage can be skipped — even hotfixes traverse the full path, just with a compressed timeline.

"A deploy that can't be described in four sentences probably shouldn't ship."

1 · Build

Container images are built reproducibly from the committed SHA. The build step produces exactly one artifact per service — a signed OCI image — and uploads it to the internal registry. Build caches are keyed by the dependency graph, not by time, which is why a clean rebuild is still fast.

# .ci/build.yaml
build:
  artifact: oci-image
  sign: cosign
  cache:
    key: "{{ deps.hash }}"

2 · Test

Tests run against the built image, not the source tree — this catches the entire class of "worked on my machine" regressions. Integration tests spin up a full dependency graph via docker compose; contract tests compare API shapes against the last-green artifact in staging.

3 · Stage

Staging is a full mirror of production — same topology, same secrets (from a separate Vault namespace), real traffic shadowed from prod at 5%. Any alert that would fire in prod will fire here first, which is the only way to earn a promotion.

4 · Promote

Promotion to production is the only stage that requires human approval. The on-call engineer gets a link, reviews the staging diff, and clicks to promote. Behind the scenes, the service runs in a dual-read window — consuming both old and new configuration simultaneously — for at least one full deploy cycle, which is usually 10–15 minutes.

This is also where secret rotation happens: the new secret is written to Vault, the service picks it up under dual-read, the old secret is revoked only after the next deploy completes cleanly.

When it breaks

A failed promotion is automatically rolled back — the previous image stays pinned, the dual-read window collapses back to the old value, and the on-call gets paged with the failing stage and a link to the build logs. No manual rollback ritual is required.
