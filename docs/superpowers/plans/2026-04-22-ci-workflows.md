# CI Workflows Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add two GitHub Actions workflows — `ci-frontend.yml` and `ci-backend.yml` — that run lint → build → test for their respective package on `push` to `main` and on pull requests, with path-filtered triggers so each workflow only runs when its package changes.

**Architecture:** Two independent workflow files under `.github/workflows/`. Each is a single job with sequential steps. Path filters on triggers scope each workflow to its own package directory plus its own workflow file. Adds `ruff` as a backend dev dependency; no ruff config file — defaults only. No changes to local tooling files, branch protection, or version-pin files.

**Tech Stack:** GitHub Actions, pnpm 10, Node 22, Python 3.13, `uv`, `ruff`.

**Source spec:** `docs/superpowers/specs/2026-04-22-ci-workflows-design.md`

## Task 3: Create the frontend CI workflow

**Files:**
- Create: `.github/workflows/ci-frontend.yml`

- [ ] **Step 1: Create the `.github/workflows/` directory if missing**

Run:

```bash
mkdir -p .github/workflows
```

- [ ] **Step 2: Write `.github/workflows/ci-frontend.yml`**

Exact file content:

```yaml
name: ci-frontend

on:
  push:
    branches: [main]
    paths:
      - 'frontend/**'
      - '.github/workflows/ci-frontend.yml'
  pull_request:
    paths:
      - 'frontend/**'
      - '.github/workflows/ci-frontend.yml'

concurrency:
  group: ci-frontend-${{ github.ref }}
  cancel-in-progress: true

jobs:
  ci:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup pnpm
        uses: pnpm/action-setup@v4
        with:
          version: 10

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: pnpm

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Lint
        run: pnpm --filter frontend lint

      - name: Build
        run: pnpm --filter frontend build

      - name: Test
        run: pnpm --filter frontend test
```

- [ ] **Step 3: Validate YAML syntax**

Run:

```bash
python3 -c "import yaml; yaml.safe_load(open('.github/workflows/ci-frontend.yml'))" && echo OK
```

Expected: `OK`.

- [ ] **Step 4: Confirm the pnpm filter command works locally**

Run (from repo root):

```bash
pnpm --filter frontend lint && pnpm --filter frontend build && pnpm --filter frontend test
```

Expected: all three commands pass sequentially. This mirrors exactly what CI will run; if it fails here, it will fail in CI.

- [ ] **Step 5: Commit**

```bash
git add .github/workflows/ci-frontend.yml
git commit -m "ci: add frontend lint/build/test workflow"
```

## Task 4: Create the backend CI workflow

**Files:**
- Create: `.github/workflows/ci-backend.yml`

- [ ] **Step 1: Write `.github/workflows/ci-backend.yml`**

Exact file content:

```yaml
name: ci-backend

on:
  push:
    branches: [main]
    paths:
      - 'backend/**'
      - '.github/workflows/ci-backend.yml'
  pull_request:
    paths:
      - 'backend/**'
      - '.github/workflows/ci-backend.yml'

concurrency:
  group: ci-backend-${{ github.ref }}
  cancel-in-progress: true

jobs:
  ci:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: backend
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup uv
        uses: astral-sh/setup-uv@v3
        with:
          enable-cache: true

      - name: Install dependencies
        run: uv sync --extra dev

      - name: Lint
        run: uv run ruff check .

      - name: Build
        run: uv run python -m compileall kb tests

      - name: Test
        run: uv run pytest
```

- [ ] **Step 2: Validate YAML syntax**

Run:

```bash
python3 -c "import yaml; yaml.safe_load(open('.github/workflows/ci-backend.yml'))" && echo OK
```

Expected: `OK`.

- [ ] **Step 3: Confirm the backend command sequence works locally**

Run (from `backend/`):

```bash
uv sync --extra dev && uv run ruff check . && uv run python -m compileall kb tests && uv run pytest
```

Expected: all four commands pass in sequence. This mirrors exactly what CI will run.

- [ ] **Step 4: Commit**

```bash
git add .github/workflows/ci-backend.yml
git commit -m "ci: add backend lint/build/test workflow"
```

## Task 5: Observational smoke tests on GitHub

**Files:** none — this task is verification against live GitHub Actions behavior.

CI workflows cannot be unit-tested. The spec defines seven observable criteria. This task walks through them.

**Pre-requisite:** The three previous tasks are committed on the current feature branch. User has explicitly asked to push and open a PR. (Per project rules, the executing agent must pause and confirm before any `git push` or `gh pr create`.)

- [ ] **Step 1: Push the branch and open a PR against `main`**

Ask the user first. Once approved, run:

```bash
git push -u origin HEAD
gh pr create --fill --base main
```

- [ ] **Step 2: Verify Criterion 2 — PR with backend-only changes triggers only `ci-backend`**

On the PR page (the just-created PR touches `backend/pyproject.toml`, `backend/uv.lock`, and `.github/workflows/*.yml`, so BOTH workflows are expected to trigger for this initial PR — this is correct because `.github/workflows/ci-frontend.yml` is in its own path filter).

Expected checks on the PR:
- `ci-frontend / ci` — runs (because `ci-frontend.yml` was added)
- `ci-backend / ci` — runs (because `ci-backend.yml` and backend files were added)

If either is missing, the trigger configuration is wrong.

- [ ] **Step 3: Verify Criterion 5 — steps run in declared order**

Open the `ci-frontend / ci` run log and confirm the step order is: Checkout → Setup pnpm → Setup Node → Install dependencies → Lint → Build → Test. Open `ci-backend / ci` and confirm: Checkout → Setup uv → Install dependencies → Lint → Build → Test.

Expected: both workflows end green.

- [ ] **Step 4: Verify Criterion 1 — a PR touching only `frontend/**` runs only `ci-frontend`**

Make a trivial frontend-only change on the PR branch (e.g., add a blank line to `frontend/README.md` or any existing frontend file):

```bash
echo "" >> frontend/src/App.tsx   # or any trivial no-op
git add frontend/src/App.tsx
git commit -m "test: trigger ci-frontend only"
git push
```

On the PR, confirm: a new `ci-frontend / ci` run starts; no new `ci-backend / ci` run starts for this push.

Revert the no-op after verification:

```bash
git revert HEAD --no-edit
git push
```

- [ ] **Step 5: Verify Criterion 2 — a PR touching only `backend/**` runs only `ci-backend`**

```bash
echo "# noqa" >> backend/kb/__init__.py
git add backend/kb/__init__.py
git commit -m "test: trigger ci-backend only"
git push
```

Confirm: new `ci-backend / ci` run starts; no new `ci-frontend / ci` run starts.

Revert:

```bash
git revert HEAD --no-edit
git push
```

- [ ] **Step 6: Verify Criterion 3 — a PR touching only root files runs neither**

```bash
echo "" >> README.md
git add README.md
git commit -m "test: root-only change should trigger neither workflow"
git push
```

Confirm: no new run starts for either workflow on this push.

Revert:

```bash
git revert HEAD --no-edit
git push
```

- [ ] **Step 7: Verify Criterion 7 — cancel-in-progress works**

Push two commits back-to-back to the PR branch on the same package:

```bash
echo "" >> frontend/src/App.tsx
git add frontend/src/App.tsx
git commit -m "test: first push for cancel-in-progress"
git push
# Immediately:
echo "" >> frontend/src/App.tsx
git add frontend/src/App.tsx
git commit -m "test: second push for cancel-in-progress"
git push
```

On the PR's Actions tab, confirm: the first `ci-frontend / ci` run shows status "Cancelled"; the second is running or completed.

Revert both:

```bash
git revert HEAD~1..HEAD --no-edit
git push
```

- [ ] **Step 8: Verify Criterion 6 — direct push to `main` triggers the right workflow**

This can only be observed after merge. After the PR is merged to `main` and a later commit lands directly on `main` touching (e.g.) `backend/kb/`, confirm that `ci-backend` runs for that `main` commit.

Mark this criterion as **deferred to first post-merge direct commit** — it cannot be verified from within the PR itself. Note in the PR description that this remains to be observed.

- [ ] **Step 9: Merge the PR when all in-PR criteria pass**

Ask the user for explicit approval before merging. Once approved:

```bash
gh pr merge --squash
```
